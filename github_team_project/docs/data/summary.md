# 📌 팀 플레이 프로젝트 요약 (버전 3)

## 🎯 핵심 목표

- Git을 개인 도구가 아닌 **협업 이력 관리 시스템**으로 사용
- 브랜치, PR, 코드리뷰, 머지 전략의 필요성 체감
- 충돌(conflict)을 자연스러운 협업 과정으로 이해

---

## 🧩 기본 규칙

- main 브랜치 직접 push ❌
- 모든 작업은 **issue 기반**
- 커밋 메시지에 **이슈 번호 포함**
- feature 브랜치에서 작업
- PR 후 최소 1명 리뷰
- pull 금지 → fetch 후 merge/rebase 판단

---

## 🏷️ 브랜치 전략

main → 항상 안정 상태
feature/이슈번호-작업명 → 작업용 브랜치

예:

feature/4-member-a-profile
feature/6-topic-restaurant

---

## 🧾 커밋 규칙

[#이슈번호] 작업 설명

예:

[#4] docs: 팀원 소개 작성
[#9] fix: README 오타 수정

---

## 🔀 PR & 리뷰

### PR 목적

- 작업을 바로 merge하지 않고 **검토 과정 경험**

### 리뷰 체크

- 이슈 연결 여부
- 커밋 메시지 규칙
- 변경 범위 적절성
- 링크 오류 여부
- 충돌 가능성

---

## 🔄 머지 전략

### 추천: squash merge

- main 히스토리 깔끔
- PR 단위 관리 용이

### 비교 학습

- merge commit → 흐름 확인
- rebase merge → 선형 히스토리

---

## ⚠️ 충돌 실습

### 방법

- README 수정 이력 테이블 공동 수정
- 같은 위치 수정 → conflict 발생

### 핵심

- 충돌은 정상적인 협업 결과
- Git이 아니라 **사람이 최종 결정**

---

## 🔁 fetch + rebase

git fetch origin
git rebase origin/main

- pull 대신 사용
- 최신 변경사항을 내 작업에 반영

---

## 🚨 공유 커밋 수정 위험성

- amend / rebase / force push는
  → **이미 공유된 이력 변경**

### 결론

공유된 커밋 수정 = 과거 이력 재작성
→ 협업 혼란 발생 가능

---

## 🧠 핵심 학습 포인트

- 브랜치 분리의 필요성
- PR & 코드 리뷰 이유
- 머지 전략 차이 이해
- 충돌 해결 경험
- fetch + rebase 흐름
- 협업에서 Git 규칙의 중요성

---

## ✅ 한줄 정리

Git 협업의 본질은
"코드를 합치는 것"이 아니라
"이력을 함께 관리하는 것"이다.
